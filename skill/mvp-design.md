# MVP Design

## Purpose

The MVP exists to validate assumptions.

It is not the first version of the final product.

The goal is learning, not perfection.

---

# MVP Framework

Ask:

What is the smallest product capable of validating the core hypothesis?

---

# Core Hypothesis

Every MVP should test one primary assumption.

Examples:

* Users want on-chain prediction markets.
* Users need better wallet analytics.
* Users will pay for portfolio automation.

If the hypothesis is unclear, the MVP is unclear.

---

# MVP Scope

Include:

* Core functionality
* Essential onboarding
* Basic analytics

Exclude:

* Nice-to-have features
* Complex customization
* Edge cases

---

# Prioritization Matrix

Must Have

* Required for value

Should Have

* Helpful but not essential

Could Have

* Nice additions

Won't Have

* Deliberately excluded

---

# MVP Metrics

Track:

* Activation
* Retention
* Engagement
* Revenue

Avoid vanity metrics.

---

# Validation Questions

After launch ask:

* Did users get value?
* Did users return?
* Did users recommend it?
* Would users be disappointed if it disappeared?

---

# MVP Checklist

☐ Core hypothesis identified

☐ Scope reduced

☐ Success metrics defined

☐ Analytics implemented

☐ Feedback system installed

Ship early.

Learn quickly.

Iterate relentlessly.
